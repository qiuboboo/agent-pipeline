# Pipeline2 标注阶段实现与结果汇报（落实版）

## 0. 汇报结论

本阶段 `pipeline2` 已经从“让模型直接给答案/CoT”推进为一条可落盘、可追踪、可审计的第二步标注流水线。它的真实主线是：

`ready 样本 -> 运行态 problem/method 结构 -> 多方法 CoT 生成与答案收敛 -> PTK 底座 -> claim 拆解 -> r_nodes 归纳 -> solution_library -> evidence_bindings -> trace_mapping_index -> 结构验证/错误记录`

这条主线落实在 `pipeline2/pipeline.py` 的三层 LangGraph 中：batch graph 负责批量题目调度，problem graph 负责单题结构化标注，method graph 负责单个解法分支的 CoT 生成、答案校验、CoT 验证和修复。核心 agent 调用、契约检查和结构化数据处理分别落在 `pipeline2/agents.py`、`pipeline2/annotation_modules.py`、`pipeline2/verification_modules.py`、`pipeline2/ready_loader.py`、`pipeline2/clients.py`。

本次重点分析的是真实输出目录：

`pipeline2/eval10_retry_fixed_outputs/`

该目录对应配置 `pipeline2/configs/eval10_retry_fixed.yaml`，配置中设置 `max_problems: 10`、`problem_retry_attempts: 10`、`continue_on_problem_error: true`、`enable_problem_structure_validation: true`。实际结果是：10 道 ready 题进入运行，4 道成功产出完整 problem bundle，6 道在 PTK foundation gate 阶段失败后进入 `problem_errors`。成功的 4 道题共生成 29 条 `p_facts`、6 条 `t_facts`、23 条 `k_atoms`、6 条合格 `claim_sequences`、94 条 claims、86 个 `r_nodes`、6 个 `solution_library` 解法族。4 个 bundle 中 3 个结构验证整体通过，1 个 bundle 保留下来但 `verification_audit.passed=false`，原因是 claim-set grounding 仍有弱点。

这个结果说明两点：第一，Pipeline2 已经能把题目推理从“文本答案”沉淀成可评测的节点图；第二，它不是无条件放行，能把“答案对但 claim grounding 不足”的方法或题目标出来，这对后续生产级标注是有价值的。

## 1. 本阶段在总项目中的位置

根据 `next_step_docs/第二步标注Pipeline详细设计.md`，第二步标注阶段不是采集阶段，也不是最终发布阶段，而是接收 `ready/` 中清洗、改写、标准化后的样本，把它们变成可验证、可评测、可回流的结构化推理对象。

本阶段输入只来自 `ready/**/*.json`，由 `pipeline2/ready_loader.py` 实现。`load_ready_problem()` 只接受 `clean_pool_entries[].pool_status == ready_for_annotation` 的题；如果题目 `requires_image=true`，还必须能解析到真实图片文件，否则直接抛出 `ReadyDataContractError`。这保证了后面所有多模态 agent 不会在“没有图片但需要图片”的情况下虚构视觉事实。

本阶段输出不是一个简单答案，而是题目级 bundle，字段包括：

- `problem_record`：题目元信息、题干、标准答案、图片路径。
- `p_facts`：图像感知事实。
- `t_facts`：题干文本条件和目标。
- `k_atoms`：可复用知识原子。
- `runtime_problem.method[]`：遵循 `main.md` 风格的方法级运行态。
- `claim_sequences`：从合格 CoT 拆出的原子 claim 序列。
- `r_nodes`：规范化后的推理节点。
- `solution_library` 和 `solution_memberships`：解法族及其必要/可选节点。
- `evidence_bindings`：节点到 P/T/K 事实和前驱节点的证据绑定。
- `trace_mapping_index`：后续评测模型轨迹时计算节点命中率的索引。
- `verification_audit`：Final CoT、claim set、node set 的结构验证报告。

这和设计文档要求一致：先经过 `main.md` 的 `problem -> method[]` 运行态结构，再进入 PTK、claim、node、solution 的结构化世界。

## 2. Pipeline2 的总体结构

### 2.1 目录与模块分工

| 模块 | 实现位置 | 实际做的事 |
| --- | --- | --- |
| 配置加载 | `pipeline2/config.py` | 用 dataclass 表示路径、运行时、阈值、模型路由；`from_yaml()` 读取 YAML；`resolve_path()` 统一相对路径。 |
| ready 装载 | `pipeline2/ready_loader.py` | 扫描 `ready/**/*.json`，过滤 `ready_for_annotation`，提取题干、答案、图片、多解分数、清洗信号。 |
| LLM 客户端 | `pipeline2/clients.py` | 用 OpenAI-compatible `/chat/completions`，强制 JSON 响应，支持图片转 base64、多端点路由、超时、usage 统计。 |
| Agent 原子能力 | `pipeline2/agents.py` | 方法规划、求解、答案判等、答案修复、CoT 验证、CoT polish、node induction、solution grouping、evidence binding、trace mapping、novelty、patch。 |
| PTK/claim 闭环 | `pipeline2/annotation_modules.py` | PTK foundation 与 claim extraction 的 critic-polish 闭环，失败可恢复进度，结构可用时可降级为 heuristic fallback。 |
| 结构验证 | `pipeline2/verification_modules.py` | Final CoT 用多模态 LLM 验证；claim set 和 node set 用结构启发式检查依赖、原子性、grounding、source/mapping 完整性。 |
| 主编排 | `pipeline2/pipeline.py` | 用 LangGraph 串起 batch/problem/method 三层状态机，并负责 stage cache、重试、输出落盘、trace eval。 |
| Prompt 管理 | `pipeline2/prompts.py` | 集中定义 planner/solver/critic/polish/PTK/claim/node/solution/evidence/trace 等 system prompt 与 user prompt builder。 |

### 2.2 三层 LangGraph

第一层是 batch graph。`build_batch_graph()` 只有一个核心节点 `process_problems_in_parallel`，它调用 `_process_problems_in_parallel_node()`，按配置 `max_problem_workers` 对题目并发处理。本次 eval 配置是 `max_problem_workers: 1`，所以是单题顺序跑，便于排查错误。

第二层是 problem graph。`build_problem_graph()` 中的顺序是：

`prepare_methods -> build_ptk -> run_methods -> extract_claims -> induce_nodes -> group_solutions -> bind_evidence -> validate_problem_structure -> finalize_problem_bundle`

这一层负责单题从“题目对象”变成“结构化 bundle”。它的好处是把题目级公共底座（PTK、r_nodes、solution_library）和方法级临时状态分开，避免把某个 CoT 的局部幻觉直接污染题目级节点库。

第三层是 method graph。`build_method_graph()` 中的顺序是：

`generate_cot -> answer_check -> verify_round_0 -> polish/verify 最多 3 轮 -> finalize_method`

这一层只处理一个 `method_id`。它先让 solver 生成 `CoT_raw` 和 `model_answer`，再用答案判等器收敛答案，之后才做 CoT 验证。这样设计的原因是：答案错误和推理链质量是两个不同问题，先把答案是否等价收敛掉，后面的 CoT verifier 才能专注判断推理是否 grounded、是否遵守方法、是否可拆成 claim。

## 3. 为什么这样设计有好处

### 3.1 先过滤 ready，再进标注，减少源头噪声

`ready_loader.py` 的 `load_ready_problem()` 不接受任意样本，只接受 `ready_for_annotation`，并且要求题干、标准答案、必要图片都存在。这相当于在进入高成本 LLM 标注前先做输入契约检查。这样做的直接好处是后续 agent 出错时更容易定位为模型输出或中间结构问题，而不是上游样本缺字段。

### 3.2 `problem -> method[]` 运行态把多解法显式化

方法数量由 `agents.target_method_count_from_score()` 实现，阈值来自配置 `method_score_thresholds: [0.33, 0.67]`：低于 0.33 生成 1 种方法，0.33 到 0.67 生成 2 种方法，高于等于 0.67 生成 3 种方法。本次四个成功样本的 `initial_multi_solution_score` 分别是 0.56、0.36、0.58、0.64，所以都规划了 2 个方法。

这样做的好处是：多解法不是事后从一段 CoT 里猜出来，而是从一开始就作为方法分支存在。每个方法有自己的 `method_draft`、`CoT_raw`、`CoT_final`、`verify_reports` 和 claim 抽取状态，后面能清楚判断哪个方法合格、哪个方法只能停留在 CoT 层。

### 3.3 答案判等先走确定性逻辑，减少无谓 LLM 判断

`agents.deterministic_answer_match()` 会先做规范化文本、数字等价、多行答案、多合法答案、公式清洗等比较。只有确定性失败时才调用 `AnswerEquivalenceJudge`。例如 `prob_016bc...`、`prob_0d655...`、`prob_35fd...` 的方法答案都走了 `deterministic_match`；`prob_1b271...` 的答案是 `19.5 cm`，标准答案是 `19.5`，因此调用了 LLM 判等，并判定“单位 cm 不改变数值正确性”。这避免了把简单数字题全部交给模型判等，也保留了单位、格式差异的弹性。

### 3.4 PTK 底座把视觉事实、文本目标、知识规则拆开

PTK 由 `annotation_modules.build_ptk_foundation()` 调用 `PerceptionExtraction`、`TextCondition`、`KnowledgeLibrarian` 产生，然后由 `PTKFoundationCritic` 审计；失败时调用 `PTKFoundationPolish` 修复，最多 2 轮。好处是：下游 claim 不再直接从题目和 CoT 里“凭感觉”抽节点，而是有一个共享的证据底座。

例如圆周长题 `prob_1b271...` 初始 PTK 被 critic 打回，原因是：

> English original: “The foundation does not explicitly ground the black dot as the circle's center. Current p2 only says the dot is near the middle, while k1 requires that the segment pass through the center to identify a diameter.”
>
> 中文翻译：底座没有明确说明黑点是圆心。当前 p2 只说黑点在中间附近，但 k1 要求线段穿过圆心才能把它识别为直径。

修复后 `p2` 变为：

> English original: “A black dot is drawn at the apparent center of the circle.”
>
> 中文翻译：圆的明显中心处画了一个黑点。

这不是文字润色，而是补上“直径识别”的必要证据前提。后续 claim 里“该线段是直径”才能依赖 `p2/p4/p6 + k1`，不会变成空推理。

### 3.5 只让合格 CoT 进入 claim 和 node，防止污染节点库

`pipeline.py` 的 `_extract_claims_node()` 明确要求：

`method.is_answer_match == true` 且 `method.is_final_CoT_qualified == true`

才会尝试 claim extraction。即使 CoT 合格，claim extraction 还必须通过 `ClaimVerify/ClaimPolish` 闭环。若 3 轮后仍不通过，该方法会记录到 `claim_extraction_failures`，不会进入 `claim_sequences`，也不会参与 `r_nodes` 和 `solution_library`。

这在两个样例中很关键：

- `prob_0d655...` 的第 2 种方法答案正确、CoT verifier 也通过，但 claim extraction 失败，因为该方法使用右角/三角形路线，而当前 PTK 底座只保留了平行线直接求角路线。
- `prob_35fd...` 的第 1 种方法答案正确、CoT verifier 也通过，但 claim extraction 失败，因为它直接断言 T/U 与 C/D 的对应关系，没有充分拆出三弧标记和多边形顺序的桥梁。

这说明 Pipeline2 的节点库不是“答案对就入库”，而是“答案对、CoT 合格、claim 可原子化、依赖和 grounding 合格”才入库。

### 3.6 节点、解法族和证据绑定让后续评测可计算

`agents.induce_nodes()` 把 claim 规范化为 `r_nodes`，并生成 `claim_mappings`。它还按 method 做 namespacing，避免不同方法都叫 `c1` 时冲突。`agents.group_solutions()` 再把节点组织成 `solution_library`，区分 `required_r_ids`、`optional_r_ids` 和 `ordered_core_path`。`agents.bind_evidence()` 用词面重叠和 claim 依赖把每个 r_node 绑定回 `p_fact_ids`、`t_fact_ids`、`k_atom_ids`、`predecessor_r_ids`，并给出 `support_strength`。

最终 `build_trace_mapping_index()` 把 node catalog 和 solution 节点集合压缩成评测索引。这样后续模型输出一条 CoT 后，可以用 `evaluate_traces()` 计算：

- `node_hit_rate_total`
- `node_hit_rate_required`
- `topology_consistency_score`
- `evidence_grounding_score`
- 是否满足新解候选阈值

也就是说，这一阶段产物不是只给人看，而是给下一阶段自动评测和回流用的。

## 4. 每一步用什么对什么做了什么

### Step 1：Ready Intake Loader

用 `pipeline2/ready_loader.py` 对 `ready/` 里的 `prob_*.json` 做扫描和契约校验。它从 `clean_problem_record`、`candidate_problem_record`、`normalized_assets`、`raw_asset_bundle`、`asset_records` 中提取题干、标准答案、图片路径、多解分数和清洗元信息。

本次成功的四题都来自 Geometry3K，`requires_image=true`，图片路径都解析到了 `ready/.../artifacts/images/..._primary.png`，`alignment_status=good`，`solvability_score=1`。

### Step 2：Runtime State Builder + Method Planner

用 `pipeline.py::_prepare_methods_node()` 对每道题构造 `runtime_problem.method[]`。方法数由 `initial_multi_solution_score` 和阈值 `[0.33,0.67]` 决定；具体方法草稿由 `agents.plan_method_collection()` 生成，最多 3 次尝试，内部用 `_method_candidates_are_duplicate()` 去掉语义重复的方案。

用 `annotation_modules.build_ptk_foundation()` 对题目图文构造 PTK 底座。它先抽 `p_facts/t_facts/k_atoms`，再由 critic 检查是否最小、是否 grounded、是否遗漏关键事实。失败后 polish，最多 2 轮。

用 `annotation_modules.extract_claims_bundle()` 对合格 CoT 做原子 claim 拆解。拆解结果必须经过 `ClaimVerify`，失败时由 `ClaimPolish` 修复，最多 2 轮。若 LLM 闭环没完全通过，但 `verification_modules.validate_claim_set()` 判断结构可用，可以用 heuristic fallback；否则方法被写入 `claim_extraction_failures`。

### Step 6：Node Induction

用 `agents.induce_nodes()` 对每个方法的 claim 做规范化，生成 `r_nodes` 和 `claim_mappings`。实现上按 method 分组调用 NodeInduction，再用 canonical claim 的规范化文本作为 group key，生成稳定 `r_id = r_<stable_digest>`。claim id 在 `_namespace_claims_for_method()` 中加上 `m{method_id}__` 前缀，解决不同方法都有 `c1/c2` 的冲突问题。

四个成功 bundle 共生成 86 个 r_nodes：

| problem_id | r_nodes | solution_library |
| --- | ---: | ---: |
| `prob_016bc...` | 26 | 2 |
| `prob_0d655...` | 5 | 1 |
| `prob_1b271...` | 30 | 2 |
| `prob_35fd...` | 25 | 1 |

### Step 7：Solution Grouper + Evidence Binder

用 `agents.group_solutions()` 把 r_nodes 组成解法族。每个 solution 有 `required_r_ids`、`optional_r_ids`、`ordered_core_path`、`supported_answer`、`member_method_ids`。然后 `agents.bind_evidence()` 把节点绑定到 PTK 事实和前驱节点。

例如 `prob_0d655...` 的唯一解法族是：

> English original: “Direct parallel-line angle transfer: identify ∠2 at F and the given 39° angle at D as the two angles formed by transversal FD with parallel segments FC and ED, then use the parallel-lines angle congruence relationship to conclude ∠2 = 39°.”
>
> 中文翻译：直接使用平行线角度迁移：识别 F 点的 ∠2 和 D 点的 39° 都是横截线 FD 与平行线 FC、ED 形成的角，再用平行线角相等关系得到 ∠2 = 39°。

这个 solution 的核心节点包括：`∠2 is the angle at F formed by rays FC and FD`、`The marked 39° angle is the angle at D formed by rays DE and DF`、`FC is parallel to ED`、`If FC ∥ ED and FD is a transversal...`、`m∠2 = 39°`。

### Step 8：Problem Structure Validation

用 `pipeline.py::_validate_problem_structure_node()` 做最终结构检查。Final CoT 仍调用多模态 LLM 验证；claim set 和 node set 由 `verification_modules.validate_claim_set()`、`validate_node_set()` 做启发式结构检查。

这里的 heuristic 不是判断数学真理，而是检查结构能不能可靠入库：claim 是否原子、依赖是否只指向已出现 claim、claim 文本是否能在 CoT 中找到 grounding token、是否有 final_answer claim；node 是否有 source_claim_ids，claim_mappings 是否覆盖，canonical_claim 是否非空。

四个成功输出里，3 个 `verification_audit.passed=true`；`prob_35fd...` 为 false，原因是 method 2 的 claim set 存在 weak grounding。这个失败被保存在 bundle 中，而不是吞掉。

### Step 9：Trace Mapping / Novelty / Patch 入口

评测与回流入口已经在 `pipeline.py::evaluate_traces()` 和 `agents.map_trace()/detect_novelty()/build_patch()` 实现。本次 `eval10_retry_fixed_outputs` 主要是建库输出，没有实际 trace eval 文件；但每个成功 bundle 已包含 `trace_mapping_index`，能给下一步模型轨迹评测使用。

## 5. 四个真实样例分析

### 5.1 `prob_016bc73dfd9342abef609367`：两条直线相交求角

题目原文：

> English original: “Find $m\\angle VXW$.”
>
> 中文翻译：求 `∠VXW` 的度数。

标准答案是 `65`。PTK 抽出了 7 条视觉事实、1 条文本目标、7 条知识原子。关键视觉事实包括：

> English original: “A straight blue segment runs from Z through X to W.”
>
> 中文翻译：一条蓝色直线段从 Z 经过 X 到 W。

> English original: “Another straight blue segment runs from Y through X to V.”
>
> 中文翻译：另一条蓝色直线段从 Y 经过 X 到 V。

> English original: “The angle at X between rays XZ and XV is ... labeled '(2x + 65)°'.”
>
> 中文翻译：X 点处由 XZ 和 XV 形成的角标为 `(2x + 65)°`。

Pipeline 规划出两种方法。

第一种方法用对顶角：

> English original: “Use the intersecting-lines rule that vertical angles are equal, so 2x+65 = 4x+15. Solve for x, then substitute back ... Finally use the straight-line/linear-pair fact that ∠ZXV and the target ∠VXW are supplementary.”
>
> 中文翻译：使用两直线相交时对顶角相等，所以 `2x+65 = 4x+15`。解出 x 后代回，再用 `∠ZXV` 与目标角 `∠VXW` 构成线性对、互补的事实。

第二种方法不用直接说“对顶角相等”，而是设 `t = m∠VXW`，分别写两个补角方程：

> English original: “Let t = m∠VXW... (2x+65) + t = 180... (4x+15) + t = 180. Since both expressions equal 180 - t, set them equal.”
>
> 中文翻译：设 `t = m∠VXW`，由两个线性对分别得到 `(2x+65)+t=180` 和 `(4x+15)+t=180`。因为两个表达式都等于 `180-t`，所以令它们相等。

效果：

- 2 个方法都答案匹配、CoT 合格。
- method 1 拆出 14 条 claims，method 2 拆出 19 条 claims。
- 归纳出 26 个 r_nodes、2 个 solution。
- `verification_audit.passed=true`。

这里能看出多解法设计的价值：同一个题，solution_library 保留了两条不同主路径。一条的核心节点是“对顶角相等 -> 解 x -> 线性对求目标角”；另一条的核心节点是“目标角 t -> 两个补角方程 -> 解 x/t”。后续评测时，如果学生或模型走其中任一路径，都可以用各自的 required nodes 做命中率计算。

### 5.2 `prob_1b2715b568fa6d8f9f722d1e`：圆周长题

题目原文：

> English original: “Find the circumference of the figure. Round to the nearest tenth.”
>
> 中文翻译：求该图形的周长，并四舍五入到十分位。

标准答案是 `19.5`。图中一条粉色线段标为 `6.2 cm`，穿过圆心并连接圆的两端。

PTK critic 第 0 轮指出底座不够严谨：

> English original: “The foundation does not explicitly ground the black dot as the circle's center... This leaves the key diameter claim under-supported.”
>
> 中文翻译：底座没有明确说明黑点是圆心，这会让“该线段是直径”这个关键判断缺少支撑。

修复后产生的关键 P/K 包括：

> English original: “The pink segment passes through the black dot.”
>
> 中文翻译：粉色线段穿过黑点。

> English original: “The pink segment has endpoints on the circle.”
>
> 中文翻译：粉色线段两个端点在圆上。

> English original: “Diameter identification: a chord passing through the center of a circle is the diameter.”
>
> 中文翻译：直径识别规则：穿过圆心的弦是直径。

Pipeline 规划出两种方法。

第一种方法直接用直径公式：

> English original: “Use the circle rule C = πd, substitute d = 6.2, compute 6.2π ≈ 19.5, and round to the nearest tenth.”
>
> 中文翻译：使用圆周长公式 `C = πd`，代入 `d = 6.2`，计算 `6.2π ≈ 19.5`，再按要求保留一位小数。

第二种方法先转半径：

> English original: “First convert this to a radius by halving: r = 6.2/2 = 3.1 cm. Then use the circumference formula in radius form, C = 2πr.”
>
> 中文翻译：先把直径除以 2 得到半径 `r = 3.1 cm`，再用半径形式的公式 `C = 2πr`。

效果：

- 2 个方法都答案匹配、CoT 合格，最后答案为 `19.5 cm`；标准答案 `19.5` 通过答案等价判断。
- method 1 拆出 14 条 claims，method 2 拆出 17 条 claims。
- 归纳出 30 个 r_nodes、2 个 solution。
- `verification_audit.passed=true`。

这个例子说明 PTK 先行的好处：如果没有 `黑点是圆心`、`线段端点在圆上`、`穿过圆心的弦是直径` 三个前提，`6.2 cm 是直径` 就只是模型猜测；现在它变成了可追踪的推理链节点。

### 5.3 `prob_0d6557c19778c9f657054b2e`：平行线求角

题目原文：

> English original: “Find $m\\angle 2$.”
>
> 中文翻译：求 `∠2` 的度数。

标准答案是 `39`。PTK 第 0 轮曾经包含多余路线，被 critic 打回：

> English original: “The k_atoms are not minimal... k5-k8 introduce speculative alternate-route knowledge that is not needed for solving the stated goal.”
>
> 中文翻译：知识原子不够最小化，k5 到 k8 引入了求解目标不需要的猜测性替代路线。

修复后，PTK 保留了平行线直接求角的必要事实：

> English original: “The upper horizontal segment and the lower horizontal segment each have one matching pink arrow marker on them.”
>
> 中文翻译：上方水平线段和下方水平线段都有相同的粉色箭头标记。

> English original: “Parallel-line arrow marks indicate the tagged lines are parallel.”
>
> 中文翻译：平行线箭头标记表示被标记的线互相平行。

method 1 走直接平行线角迁移：

> English original: “Use FD as a transversal across the parallel lines FC and ED, so the acute angle it makes with FC at F equals the acute angle it makes with ED at D. Therefore m∠2 = 39°.”
>
> 中文翻译：把 FD 作为截线，穿过平行线 FC 和 ED，所以它在 F 处与 FC 形成的锐角等于在 D 处与 ED 形成的锐角。因此 `m∠2 = 39°`。

method 2 走右角/三角形路线，CoT verifier 通过，但 claim extraction 失败。失败原因很有代表性：

> English original: “The sequence is not grounded by the provided PTK foundation... it relies on visual facts such as the left vertical A-F-E and right-angle markers at F, E, and C that are not supplied in the P facts.”
>
> 中文翻译：这条 claim 序列没有被当前 PTK 底座支撑。它依赖 A-F-E 左侧竖线以及 F、E、C 处直角标记等视觉事实，但这些事实没有出现在 P facts 中。

效果：

- 2 个方法都答案正确、CoT 合格。
- 只有 method 1 进入结构化，拆出 5 条 claims、5 个 r_nodes、1 个 solution。
- method 2 被记录在 `claim_extraction_failures`，没有污染 `r_nodes`。
- `verification_audit.passed=true`，因为进入结构化的 method 1 通过了最终验证。

这个例子说明当前系统的质量边界：它允许多方法 CoT 存在，但结构化入库必须和 PTK 底座一致。method 2 不是数学上一定错，而是“当前 PTK 没支撑这条路线”，所以暂时不能沉淀为 ground truth 节点。

### 5.4 `prob_35fd3bf5c4d60d72e63599e3`：相似多边形求 UT

题目原文：

> English original: “The two polygons are similar. Find UT.”
>
> 中文翻译：两个多边形相似，求 UT。

标准答案是 `22.5`。PTK critic 第 0 轮把一个重要问题指出来了：P 层不应该提前说“对应”，只应该描述看见了什么。

> English original: “p10 is slightly too inferential for a perception fact because it states angle-mark 'matches'. The visual fact should only report the arc counts at each labeled vertex, leaving correspondence as downstream reasoning.”
>
> 中文翻译：p10 对感知事实来说推断性太强，因为它说了角标记“匹配”。视觉事实应该只报告每个顶点有几条弧线，把“对应关系”留给后续推理。

修复后的 p10 是纯视觉事实：

> English original: “The visible angle-mark arc counts are: on the larger polygon, R has 1 arc, S has 2 arcs, T has 3 arcs, U has 3 arcs, and V has 4 arcs; on the smaller polygon, A has 1 arc, B has 2 arcs, C has 3 arcs, D has 3 arcs, and E has 4 arcs.”
>
> 中文翻译：可见角标记弧数为：大多边形中 R 为 1 弧，S 为 2 弧，T 为 3 弧，U 为 3 弧，V 为 4 弧；小多边形中 A 为 1 弧，B 为 2 弧，C 为 3 弧，D 为 3 弧，E 为 4 弧。

method 1 直接用比例：

> English original: “Since the polygons are similar, the scale factor from small to large is 18/4 = 4.5. Apply that scale factor to CD: UT = 5 × 4.5 = 22.5 m.”
>
> 中文翻译：因为两个多边形相似，从小图到大图的比例因子是 `18/4 = 4.5`，把它乘到 CD 上：`UT = 5 × 4.5 = 22.5 m`。

但 method 1 的 claim extraction 失败，原因是它没有把重复的三弧顶点消歧讲清楚：

> English original: “Because both T and U on the large polygon and both C and D on the small polygon have 3 angle-mark arcs, c13 and c14 cannot be justified ... without an explicit bridge about preserving cyclic order or adjacency.”
>
> 中文翻译：因为大多边形的 T、U 和小多边形的 C、D 都是三弧标记，如果没有一个关于循环顺序或邻接关系保持的显式桥梁，就不能充分证明 `T ↔ C` 和 `U ↔ D`。

method 2 使用未知量标签直接列比例：

> English original: “Set up a proportion with corresponding sides: 18/4 = (y + 2)/5. Solve to get y + 2 = 22.5, so UT = 22.5 m.”
>
> 中文翻译：用对应边列比例：`18/4 = (y + 2)/5`。解得 `y + 2 = 22.5`，所以 `UT = 22.5 m`。

效果：

- 2 个方法都答案正确、CoT 合格。
- method 1 claim extraction 失败；method 2 进入结构化，拆出 25 条 claims、25 个 r_nodes、1 个 solution。
- node set 结构验证通过，但整体 `verification_audit.passed=false`，原因是 method 2 的 claim set 仍有两个 weak grounding claim。

这个例子特别能说明当前 Pipeline2 的价值：它既能把可用路线沉淀下来，也能把“仍需改进的 grounding 缺口”暴露出来。最终 bundle 没有被简单丢弃，而是保留了结构产物和失败原因，方便下一轮修 PTK 或 claim prompt。

## 7. 当前不足与下一步建议

1. PTK gate 是当前主要失败点。6/10 题在 PTKFoundationGate 失败，说明 critic-polish 虽然严格，但还需要更稳定的 PTK prompt、更多启发式保底，或对复杂图形增加可接受的 degraded path。

2. CoT 合格不等于 claim 可入库。`prob_0d655...` method 2 和 `prob_35fd...` method 1 都展示了这个问题。下一步可以在 method planning 时把 PTK 底座反馈给 planner，减少 planner 生成 PTK 不支持的路线。

3. `prob_35fd...` 保留了一个结构验证未完全通过的 bundle。因为配置 `fail_on_problem_structure_validation: false`，它没有被丢弃。这对实验阶段合理，但生产阶段需要明确分级：`passed=true` 才进主库，`passed=false` 进待修队列。

4. 当前 `evidence_bindings` 是启发式词面匹配和依赖继承，速度快，但对几何符号、同义表达和图形局部关系仍可能弱。后续可把 node-level grounding 引入一个更轻量的 verifier，而不是完全依赖 lexical overlap。

5. Trace mapping / novelty / patch 已有入口，但本次 eval10_retry_fixed 主要跑建库，没有跑新 trace。下一步应拿这 4 个 bundle 作为 canonical library，喂入外部模型 CoT，真正测试 `node_hit_rate_total`、`node_hit_rate_required` 和新解候选触发。

## 8. 可以向老师强调的核心价值

Pipeline2 的核心价值不是“模型解对了几道题”，而是建立了一个可解释、可审计、可扩展的自动标注系统：

第一，它把图像事实、题干条件、知识规则、推理步骤、最终答案拆成不同层，避免把大段 CoT 当作不可检查的黑盒。

第二，它用 method graph 把答案正确性和 CoT 质量分开验证，再用 claim/node 结构化门控决定是否入库，这比“答案正确就保留 CoT”更可靠。

第三，它能沉淀多解法。`prob_016bc...` 和 `prob_1b271...` 都产出了两个可区分 solution，后续评测不会只承认一种标准路线。

第四，它能诚实暴露失败。`prob_0d655...` 的第二路线和 `prob_35fd...` 的第一路线都没有因为答案正确而被硬塞进节点库；`prob_35fd...` 的整题也因为 weak grounding 被标记为未完全通过。这种失败记录对后续改进 prompt、PTK 底座和评测规则很关键。

第五，它已经具备下一阶段节点命中率评测所需的结构：每个成功 bundle 都有 `trace_mapping_index`，每条 solution 都有 required/optional/core path 节点，后续可以直接比较模型 CoT 是否命中 ground truth 节点，而不是只看最终答案。
