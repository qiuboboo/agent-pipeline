## 4. 具体代码修改汇总

下面列出本轮对核心实现做过的修改类型。

### 4.1 `pipeline2/config.py`

改动方向：

- 增加 OpenAI 风格端点字段支持：
  - `provider`
  - `wire_api`
  - `requires_openai_auth`
  - `disable_response_storage`
- 默认模型切到：
  - `base_url = https://www.msutools.cn`
  - `model = gpt-5.4`
- 后续进一步收紧：
  - 默认不再允许 `models.fallback`
  - 配置中若仍显式启用 fallback，直接报错
- 将 `fail_on_problem_structure_validation` 默认改为更严格的 `true`

### 4.2 `pipeline2/clients.py`

改动方向：

- 新增 `responses` 风格 payload 构造。
- 新增 `responses` 响应文本抽取逻辑。
- 统一支持：
  - 纯文本
  - 多模态图文输入
- 去掉主备自动路由切换，只保留单 `primary`。
- 保留同 `primary` 端点内的重试机制。
- 增强同端点内部重试与超时窗口，目标是：
  - 允许处理外部接口抖动
  - 不引入备用模型/备用答案

### 4.3 `pipeline2/verification_modules.py`

改动方向：

- 重写 claim-set grounding heuristic：
  - 不再只依赖 claim 文本前几个 token 是否出现在 CoT 中
  - 改成综合：
    - `evidence_hint`
    - `p_facts / t_facts / k_atoms` overlap
    - `CoT` overlap
    - `standard_answer` overlap
- 使得很多原本“结构上可用、但被误判”的 claim 能被正确判定通过

### 4.4 `pipeline2/annotation_modules.py`

改动方向：

- 增强 `p_facts / t_facts / k_atoms` 的对象/列表容错解析。
- 强化 `_derive_text_facts_from_question(...)`：
  - 能处理陈述式 givens
  - 能处理填空式 goal
  - 能识别中文目标词
- 强化 `_sanitize_t_facts(...)`：
  - goal-like `given` 自动纠正为 `goal`
  - 语义重复 givens 去重
- 删除 PTK 内容级 fallback：
  - 不再因为 heuristic 看起来可用就放过
  - PTK 不合格就直接 `PTKFoundationGate`
- 删除 claim 内容级 fallback：
  - claim 审不过就直接失败
  - 不再允许拿 partial claims 继续 node induction
- 将 PTK 和 claim repair 轮数从 `2` 提高到 `4`，给严格模式更多修复机会

### 4.5 `pipeline2/agents.py`

改动方向：

- 增补 node/claim type alias：
  - `rule -> knowledge_call`
  - `task -> text_condition`
  - `premise -> text_condition`
  - `condition -> text_condition`
- 调整 agent 错误信息，去掉 `fallback_last_error` 文案
- 保持“只报告 primary 端点错误”

### 4.6 `pipeline2/pipeline.py`

改动方向：

- 删除 PTK 失败后的 degraded foundation 继续执行逻辑。
- 删除 claim 提取失败后用 partial claims 继续往下游跑的逻辑。
- 现在任何 qualified method 的 claim 提取失败，都会导致题目显式失败。
- 将 PTK / claim repair 轮数提高为 `4`
- 保留 checkpoint / stage cache 恢复执行能力，但不再允许“恢复错误结果”

### 4.7 `pipeline2/prompts.py`

改动方向：

- 强化 `CLAIM_EXTRACTION_SYSTEM_PROMPT`
- 强化 `CLAIM_POLISH_SYSTEM_PROMPT`

新增约束包括：

- 禁止一个 claim 打包多个独立观察
- 禁止把桥接规则和结论写在同一个 claim
- 测试样例推理必须拆成多条 atomic claims
- 电路题必须先有“可见结构 -> 子表达式 -> 分类”分层
- 禁止引入 verified CoT 中没有的额外对比支路
- 每条 `knowledge_call` 都必须可回指到 PTK 中已有 `k_atom`

### 4.8 `scripts/run_pipeline2_launcher.sh`

改动方向：

- 去掉 `ANNOTATION_FALLBACK_*` 的默认注入逻辑
- `run_meta.json` 中不再记录 fallback base URL

## 9. 严格模式下发现的核心问题

### 9.1 claim audit fallback 风险

在“去 fallback”之前，出现过大量：

- `claim_audit.passed = false`
- 但 `heuristic_report.pass = true`
- 最终题目级仍然继续往下游跑

这被定性为：

> 输出可用，但 claim 图谱中间质量不够干净，会给以后埋雷。

因此后续已全部去除。

### 9.2 `PTKFoundationGate` 失败

在 strict 模式下，比较高频的一类失败是：

- `PTK` 文本事实不完整
- `t_facts` 被拆成碎片
- `goal/given/constraint` 标注混乱
- 带有编号碎片、重复 givens、误把目标句当 given

## 10. 针对 EEE-Bench 做过的专项修复

为了提高 `EEE-Bench` 严格通过率，本轮做过一批很有针对性的工作。

### 10.1 失败模式识别

早期 `EEE-Bench` 容易卡在：

- `PTKFoundationGate`
- `ClaimExtractionGate`
- `method 2` claim 图过于复杂、依赖桥不充分
- `claim` 中混入 CoT 未使用的额外对比支路

### 10.2 修复手段

- 把 strict 配置改成单方法优先，避免 `method 2` 把整题拖死。
- 强化 `claim extraction / claim polish` prompt：
  - 禁止引入 verified CoT 之外的额外对比支路

  - 要求电路题先做：
    - 可见结构
    - 中间布尔表达式/子网功能
    - 分类
    - 最终答案
- 修正 `t_facts` 清洗逻辑，减少 `EEE-Bench` 中目标句被误标或 givens 不完整的问题。

## 12. 已形成的当前工程原则

经过本轮反复讨论与实现，当前已经明确建立了这些原则：

1. **不允许内容级 fallback**
   - 不允许 heuristic 假过
   - 不允许 partial claims 继续往下游
2. **不允许主备模型切换**
   - 不换备用模型
3. **允许同一 primary 端点内的基础设施级重试**
   - 只为解决 `502/503/SSL EOF/RemoteDisconnected`
4. **允许 stage cache + checkpoint 恢复执行**
   - 但不允许恢复坏结果继续跑
5. **失败必须显式暴露**
   - 不能“看起来通了，实际上藏问题”
