## 3. 各对象给标注阶段的用途

### 3.1 `problem_main_record`
- 角色：题目主索引，给下游快速判断这题是否能直接进标注。
- 关键字段：
  - `problem_id`：内部稳定主键。
  - `source_dataset` / `source_problem_id`：回溯到原数据源。
  - `raw_question_text` / `normalized_question_text`：原题干与标准化题干。
  - `raw_answer_text` / `normalized_answer_text`：原答案与标准化答案。
  - `clean_decision`：`pass/review/reject`。
  - `annotation_ready`：是否允许直接进入标注。
  - `qa_precheck_ready`：是否具备进入下一轮 QA 预检的基础。
  - `rewrite_strategy` / `open_variant_count`：本题开放化改写方式及变体个数。

### 3.2 `asset_records`
- 角色：记录题干文本、标准化文本、原图、ROI 裁剪图、开放化题干文本等资产。
- 标注时用途：如果标注员需要复核图片、题干裁剪区域或开放题文本资产，直接从这里取 `storage_uri` 与 `asset_role`。

### 3.3 `node_records`
- 角色：把题干、答案、清洗决策等拆成节点，为后续推理链标注和 QA 留接口。
- 标注阶段当前可选用，不是首要入口。

### 3.4 `cleaning_records`
- 角色：记录一次清洗决策的动作日志。
- 标注时重点看：
  - `decision` / `decision_reason_codes`
  - `rewrite_summary`
  - `review_ticket_id`
- 这决定该题是否要进人工审核队列。

### 3.5 `alignment_records`
- 角色：记录图文对齐打分。
- 对图像题尤其重要；若 `alignment_status` 差或 `consistency_score` 低，标注前应先审。

### 3.6 `field_audit_records`
- 角色：记录字段变更前后值。
- 用途：当标注员质疑题干被清洗得过度或不足时，可以直接回看 `before_value/after_value`。

### 3.7 `rewrite_reports`
- 角色：记录开放化改写策略、理由、丢弃原因，以及所有开放题变体。
- 它是“为什么这样改写”的解释层。

### 3.8 `open_ended_problem_variants`
- 角色：下游标注最核心的直接输入。
- 每条记录对应一个可标注的开放题版本，关键字段有：
  - `open_variant_id`：变体主键
  - `parent_problem_id`：回连原题
  - `variant_index`：同一题的第几个开放变体
  - `rewritten_question_text`：开放题题面
  - `expected_answer_type`：期望答案形式
  - `expected_answer`：参考答案
  - `split_role`：`single` / `part_i` 等拆分角色

## 4. 标注阶段建议怎样接

### 4.1 直接可入标注池
- 条件：`problem_main_record.annotation_ready == true` 且 `clean_decision == "pass"`。
- 推荐主输入：`records/open_ended_problem_variants.jsonl`。
- 推荐附带读取：同题 `problem_main_record` 与 `rewrite_report`。

### 4.2 需要人工审核后再标注
- 条件：满足以下任一项即可进审核队列：
  1. `clean_decision in {review, reject}`
  2. `rewrite_strategy == "split_open"`
  3. 任一开放变体 `expected_answer_type == "unknown"`
  4. 改写文本明显异常，如 `text`、`bar`、空串、极短字符串
- 这部分题在 [`open_ended_rewrite_review_report.md`](outputs/multidataset_cleaning/run_ef0fb86b3dfaed19/open_ended_rewrite_review_report.md) 中已经逐条标明。

### 4.3 当前不应进入标注的题
- `clean_decision == "reject"` 且没有恢复计划的题。
- 典型例子：Geometry3K 中大量 `rewritten_question_text = text/bar` 的样本，这类样本目前应留在修复队列，而不是标注队列。

## 5. 对“拆分之后的结果需要我审核吗”的明确回答

- 不是所有拆分题都必须由你逐条重审。
- 但 **所有 `split_open` 样本都建议人工审核一次**，原因是拆分会改变题目粒度，需要确认：
  1. 每个子题是否仍然自洽；
  2. 子题之间是否有信息遗漏；
  3. `expected_answer` 是否真的只对应当前子题；
  4. 是否存在把多目标答案错误地塞进单个子题的问题。
- 如果某条样本本身还被打成 `review/reject`，那就是“必须审核”，不只是“建议审核”。

## 7. 建议下游实际使用顺序

1. 先读取总表 `summary.json`，确认本轮哪些数据集可用。
2. 对每个可用数据集，先加载 `records/problem_main_records.jsonl`，筛出 `annotation_ready=true` 的题。
3. 再按 `problem_id` 关联 `records/open_ended_problem_variants.jsonl`，把开放题变体送入标注平台。
4. 若 `rewrite_strategy=split_open` 或 `review_ticket_id != null`，先进入人工审核池，再放行给标注员。
5. 如需追溯清洗动作，关联 `cleaning_records.jsonl` 与 `field_audit_records.jsonl`。
6. 如需核查图片问题，回看 `asset_records.jsonl` 中原图与 ROI 裁剪资产。
