## 4. candidate intake 主线接入情况

目前 candidate intake 逻辑已经接到 reference 主线中，基本链路为：

- `connector`
- `prompt extraction`
- `asset registry`
- `initial scoring`
- `candidate registrar`
- `cleaning`

### 4.1 asset registry

目前做的事情：
- 为每条样本登记：
  - question 是否存在；
  - answer 是否存在；
  - image_paths 是否存在；
  - image 是否能解析到本地实体；
  - 基本 manifest 信息。

### 4.2 initial scoring

目前做的事情：
- 给样本生成初步分数：
  - `image_dependency_score`
  - `multi_step_score`
  - `verifiability_score`
- 并记录：
  - `risk_flags`
  - `score_evidence`

### 4.3 candidate registrar

目前做的事情：
- 给样本做前置决策：
  - `keep`
  - `low_priority`
  - `reject`

### 4.4 分流逻辑已接上

此前的问题是：
- `candidate_registrar=reject` 只是记录下来；
- cleaning 仍然继续执行；
- 导致出现“intake reject，但 cleaning pass”的逻辑不一致。

现在已修复为：
- 如果 `candidate_registrar=reject`：
  - 直接走 candidate intake reject 快速路径；
  - 不再进入正常 rewrite / cleaning 主流程；
  - `problem_main_record` 中会体现为最终 reject；
  - `reject_records` 中 `stage` 会标记为 `candidate_intake`；
  - `rewrite_strategy` 会写成 `candidate_reject`。

这一步已经生效，并通过 `all_available_smoke_v2` 验证。

## 5. intake policy 的两轮状态

### 5.1 第一版（过严）

在第一版 candidate intake 分流刚接上后，问题是：
- 对缺图样本过于严格；
- 几乎把大部分数据集都在 intake 阶段挡掉；
- 尤其 `cmm_math_local` 这种文本和选项完整、图像不强依赖的数据也被挡掉。

### 5.2 当前已放宽（第二版）

目前已做的放宽：

1. **asset registry 不再默认强制“必须有图才 pass”**
   - 若 `force_requires_image=true`：缺图仍视为关键问题；
   - 若 `force_requires_image=false`：文本完整即可 pass registry；缺图仅记录 issue。

2. **candidate registrar 对文本完整但缺图的样本放宽**
   - 若：
     - question 存在；
     - answer 存在；
     - 图像缺失；
     - 但图像依赖分数不高；
   - 则不再直接 `reject`，而是给：
     - `low_priority`

这一步已经通过新的代表集测试验证有效。
